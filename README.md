# universal-go-backend

## Overview

A production-ready Go backend template designed to be cloned as the starting
point for any new project. It encodes every structural decision you'd otherwise
spend days debating: layered architecture, type-safe SQL, file-based migrations,
annotation-driven API docs, live reload, and graceful shutdown — all wired
together with zero magic, so you can read any file and understand exactly what
it does.

## Architecture: the layered pattern

Every feature follows the same vertical slice. Data flows in one direction;
dependencies point inward toward the domain.

```
HTTP Request
     │
     ▼
┌──────────────────────────────────────────────────────────┐
│  handler/article.go                                      │
│  • Decode JSON body                                      │
│  • Validate with validator/v10                           │
│  • Call service method                                   │
│  • Write JSON response / error envelope                  │
└──────────────────────┬───────────────────────────────────┘
                       │ calls
                       ▼
┌──────────────────────────────────────────────────────────┐
│  service/article.go                                      │
│  • Business rules (auth checks, quota, computed fields)  │
│  • NO SQL / NO pgx imports                               │
│  • Delegates persistence to the repository               │
└──────────────────────┬───────────────────────────────────┘
                       │ calls
                       ▼
┌──────────────────────────────────────────────────────────┐
│  repository/article.go                                   │
│  • Owns pgxpool.Pool and sqlc Queries                    │
│  • Manages transactions (Begin → writes → Commit)        │
│  • Converts pgtype.UUID / pgtype.Timestamptz → Go types  │
│  • Wraps all errors with pkg/errors context              │
└──────────────────────┬───────────────────────────────────┘
                       │ uses
                       ▼
┌──────────────────────────────────────────────────────────┐
│  db/generated/  (sqlc output)                            │
│  • Type-safe Go functions generated from SQL queries     │
│  • Never written by hand                                 │
└──────────────────────┬───────────────────────────────────┘
                       │
                       ▼
               PostgreSQL 16
```

**domain/** is shared by all layers. It contains only structs and sentinel
errors — zero logic, zero imports from the project.

## Quick start (local dev)

```bash
# 1. Clone the repository
git clone <repo-url> my-app
cd my-app

# 2. Set up environment variables
cp .env.example .envrc
direnv allow           # loads all DB_* and APP_* vars into your shell

# 3. Download Go dependencies
go mod tidy

# 4. Start the Postgres container
make docker-up

# 5. Apply the database schema  ← THE ONLY TIME migrations run
make migrate-up

# 6. Generate Swagger docs from annotations
make swagger

# 7. Start the server with live reload
make dev

# 8. Open the interactive API docs
open http://localhost:8080/swagger/index.html
```

> **Migration policy (strict):** Steps 4, 5, and 7 are always separate explicit
> commands. `make dev` never triggers migrations. Air never triggers migrations.
> The server only *verifies* the DB is reachable on startup — it never modifies
> the schema.

## Swagger / API docs

Swagger documentation is generated from Go source annotations using
[swaggo/swag](https://github.com/swaggo/swag).

**How it works:**

1. You write `// @Summary`, `// @Param`, `// @Success` comments above each handler.
2. Run `make swagger` — swag reads the annotations and writes `docs/`.
3. The app imports `_ "my-app/docs"` which calls `docs.init()` on startup,
   registering the spec with the swag library.
4. `http-swagger` serves the Swagger UI at `GET /swagger/*` using that spec.

**After adding a new handler:**

```bash
# Write your @Summary/@Param/@Success annotations above the handler function.
make swagger     # regenerates docs/
make dev         # restart or let air pick it up
# Visit http://localhost:8080/swagger/index.html
```

**Annotation reference** (add above every handler):

```go
// CreateFoo godoc
//
//	@Summary		Short description
//	@Description	Longer description
//	@Tags			foo
//	@Accept			json
//	@Produce		json
//	@Param			request  body      domain.CreateFooRequest  true  "payload"
//	@Success		201      {object}  domain.FooResponse
//	@Failure		400      {object}  domain.ErrorResponse
//	@Failure		500      {object}  domain.ErrorResponse
//	@Router			/api/v1/foos [post]
```

> **Never manually edit files inside `docs/`.** They are overwritten on every
> `make swagger` / `make build` run.

## Running in Docker

Bring up the full stack (Postgres + app) with a single command:

```bash
# Build the image and start both services
make docker-up

# View logs
docker-compose logs -f app

# Tear down (data volume is preserved)
make docker-down
```

The app service waits for Postgres to pass its health check before starting,
so there is no race condition on first boot.

> **Important:** migrations are NOT run by docker-compose. After `make docker-up`
> you must still run `make migrate-up` (pointing at the container's port) or
> exec into the container and run it there.

## Database migrations

Migrations are managed by [golang-migrate](https://github.com/golang-migrate/migrate)
using plain SQL files in `internal/db/migrations/`.

```
POLICY: Migrations never run automatically.
Always run make migrate-up manually.
```

**Create a migration:**

```bash
make migrate-create name=add_tags_to_articles
# Creates:
#   internal/db/migrations/003_add_tags_to_articles.up.sql
#   internal/db/migrations/003_add_tags_to_articles.down.sql
```

**Apply all pending migrations:**

```bash
make migrate-up
```

**Roll back the last migration:**

```bash
make migrate-down
```

## Code generation (sqlc)

sqlc turns SQL into type-safe Go functions. The workflow is:

1. Write a query in `internal/db/queries/` with a sqlc annotation:

```sql
-- name: GetUserByEmail :one
SELECT * FROM users WHERE email = $1;
```

2. Run `make sqlc` — sqlc reads the annotation, the schema, and writes Go to
   `internal/db/generated/`.

3. Call the generated function in your repository:

```go
user, err := q.GetUserByEmail(ctx, email)
```

Never write manual SQL in handler, service, or repository files. All SQL lives
in `internal/db/queries/`.

## Adding a new feature (the full pattern)

Follow these steps to add a hypothetical **users** feature. Every future
feature follows exactly this pattern.

```
Step 1: Create migration
        internal/db/migrations/003_create_users.up.sql
        internal/db/migrations/003_create_users.down.sql

Step 2: Apply the migration
        make migrate-up

Step 3: Write SQL queries
        internal/db/queries/user.sql
        -- name: CreateUser :one ...
        -- name: GetUserByID :one ...

Step 4: Regenerate sqlc
        make sqlc
        # → internal/db/generated/user.sql.go

Step 5: Create domain structs
        internal/domain/user.go
        type CreateUserRequest struct { ... }
        type UserResponse struct { ... }

Step 6: Create repository
        internal/repository/user.go
        type UserRepository struct { pool *pgxpool.Pool }
        func (r *UserRepository) CreateUser(...) (domain.UserResponse, error)
        func (r *UserRepository) GetUserByID(...) (domain.UserResponse, error)

Step 7: Create service
        internal/service/user.go
        type UserService struct { repo *repository.UserRepository }
        // Business logic only — no pgx/sqlc imports here.

Step 8: Create handler (with swaggo annotations)
        internal/handler/user.go
        // @Summary   Create a user
        // @Router    /api/v1/users [post]
        func (h *UserHandler) CreateUser(w http.ResponseWriter, r *http.Request)

Step 9: Register routes
        internal/routes/routes.go
        r.Post("/users", userHandler.CreateUser)
        r.Get("/users/{id}", userHandler.GetUserByID)

Step 10: Regenerate Swagger docs
         make swagger

Step 11: Start dev server
         make dev
         # Visit http://localhost:8080/swagger/index.html
```

## Graceful shutdown

On `SIGINT` or `SIGTERM` the server:

1. Logs the received signal.
2. Stops accepting new HTTP connections.
3. Waits up to **30 seconds** for in-flight requests to complete.
4. Closes the database connection pool.
5. Exits with code `0`.

The 30-second window is defined as `server.ShutdownTimeout` and is reused
both in `main.go` and in the server package so there is a single source of
truth.
